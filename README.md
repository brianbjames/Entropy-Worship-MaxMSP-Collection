# ☍ ENTROPY ENGINE ☍  
### a Max/MSP repository for systems that decay, drift, and dissolve

---

## ☍ Invocation

This repository is not a tool.  
It is a process.

It does not *generate* sound — it **unravels it**.  
It does not *control* structure — it **lets structure collapse**.

If you are looking for stability, leave now.  
If you are looking for repetition, you will be disappointed.  
If you are looking for control... you are already losing it.

---

## 🜃 What Lives Here

Inside this system:

- Signals degrade over time  
- Clocks drift out of sync  
- Feedback consumes its source  
- Order emerges briefly, then disappears  

Every patch is a **temporary agreement with chaos**.

---

## 🜂 Core Principles

### 1. Entropy is not a bug
Every glitch, dropout, or distortion is part of the system’s truth.

### 2. No state is permanent
Parameters are not fixed — they **wander**, **leak**, and **mutate**.

### 3. Feedback is sacred
Outputs must find their way back inward.  
Systems that do not listen to themselves cannot decay properly.

### 4. Time is elastic
Metro is a suggestion. Phase is unstable. Synchronization is fragile.

---

## 🜁 Usage

There are no instructions.

But if you insist:

1. Open a patch  
2. Turn something on  
3. Change something slowly  
4. Then stop trying to control it  

Let the system reveal its failure modes.

---

## 🜄 Recommended Practices

- Use **low-frequency modulation** to destabilize structure  
- Introduce **noise** into control signals  
- Allow **feedback loops** to reach uncomfortable thresholds  
- Avoid hard resets — let systems **decay naturally**  
- Record everything — the best moments are irreversible  

---

## 🜏 Warnings

- Patches may become unstable  
- Audio may clip, distort, or vanish  
- CPU usage may spike unpredictably  
- Results are non-repeatable by design  

This is expected. This is desired.

---

## 🜃 Structure of the Repository

```

/recordings     → artifacts of previous collapses  
```
